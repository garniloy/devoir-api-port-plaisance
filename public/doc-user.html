Users

Manage marina staff and operator accounts — register, list, retrieve, update credentials, and delete users.

Base URL /users
🔒
Security sensitive resource. User passwords are hashed with bcrypt (cost factor 10) before storage. Plain-text passwords are never persisted or returned. Treat all user endpoints as protected routes in production.

Data Model

User documents are stored in MongoDB. Passwords are always stored as bcrypt hashes — they are never returned in any response.

User MongoDB / Mongoose
FieldTypeConstraintsDescription
_id ObjectId auto MongoDB unique identifier, generated automatically.
name String required Full display name of the user.
email String required unique Email address. Acts as the primary identifier for route lookups (:email). Must be unique across all users.
password String required Bcrypt hash of the user's password. Never returned in responses.
createdAt Date auto Timestamp of document creation, managed by Mongoose.
updatedAt Date auto Timestamp of last modification, managed by Mongoose.
ℹ️
Response projection: List and single-user GET endpoints explicitly project name and email only — password and __v are stripped from all read responses. Registration and update endpoints return id, name, and email.

Validation Rules

  • All three fields (name, email, password) are mandatory on registration. Missing any one causes a Mongoose validation error and a 400 response.
  • email must be unique. Attempting to register with an already-used address returns 400 with the Mongoose duplicate-key error message.
  • password is hashed with bcrypt.hash(password, 10) before any database write — both on creation and on update. You must always send the raw plain-text password; the API handles hashing internally.
  • Routes use :email as the URL identifier — not _id. Always URL-encode email addresses that contain special characters (e.g. user%40example.com).
  • On update, only the fields present in the body are applied. name, email, and password can all be updated independently. Omitting a field means it is not touched.
  • Changing a user's email via the update endpoint changes their URL identifier for all future requests — update stored references accordingly.
  • The getAll endpoint returns a 404 when the users collection is entirely empty, but does not stop execution before the subsequent 200 send — see the real-world note on the List endpoint.

Endpoints

POST /users Register a new user ▼

Creates a new user account. The password is hashed with bcrypt before being stored. The response never includes the password hash.

Request Body

FieldTypeRequiredDescription
name string required Full name of the user.
email string required Unique email address.
password string required Plain-text password. Hashed server-side before storage.
Request example
JSON Body
{
  "name":     "Alice Martin",
  "email":    "alice@marina.fr",
  "password": "s3cur3P@ssword"
}
● 201 Created

Returns only id, name, and email. Password hash is never included.

JSON
{
  "id":    "664c3e1a2f7d8b001e0a1b2c",
  "name":  "Alice Martin",
  "email": "alice@marina.fr"
}
● 400 Bad Request

Returned when a field is missing or the email already exists (duplicate key error).

JSON
{
  "issue":   "Bad body request",
  "details": "E11000 duplicate key error collection: marina.users index: email_1 dup key: { email: \"alice@marina.fr\" }"
}
⚠️
Real-world note: There is currently no email format validation at the application layer — Mongoose only enforces uniqueness and presence. Consider adding a regex validator or a library like validator.js to reject malformed email addresses before they reach the database.
GET /users List all users ▼

Returns all user accounts projected to name and email only. Password hashes are never returned.

Request example
HTTP
GET /users
● 200 OK
JSON
[
  {
    "_id":   "664c3e1a2f7d8b001e0a1b2c",
    "name":  "Alice Martin",
    "email": "alice@marina.fr"
  },
  {
    "_id":   "664c3e1a2f7d8b001e0a1b2d",
    "name":  "Bob Leclerc",
    "email": "bob@marina.fr"
  }
]
● 404 Not Found

Sent when the collection is empty — but see the note below.

JSON
{
  "issue":   "Database error",
  "details": "no user found"
}
● 500 Internal Server Error
JSON
{
  "message": "Internal server error",
  "error":   "..."
}
🚨
Known behaviour: When the collection is empty, the service sends a 404 JSON body but does not return early — execution continues and a second res.status(200).json(users) call fires immediately after. This causes a "headers already sent" Node.js warning. Clients will typically receive the 404 body. This should be fixed by adding a return before the 404 send.
GET /users/:email Get a single user ▼

Returns one user document by email address, projected to name and email.

Path Parameters

ParameterTypeRequiredDescription
email string required The user's email address. URL-encode @ as %40.
Request example
HTTP
GET /users/alice%40marina.fr
● 200 OK
JSON
{
  "_id":   "664c3e1a2f7d8b001e0a1b2c",
  "name":  "Alice Martin",
  "email": "alice@marina.fr"
}
● 404 Not Found
JSON
{
  "issue":   "Database error",
  "details": "User not found"
}
● 500 Internal Server Error
JSON
{
  "message": "Internal server error",
  "error":   "..."
}
⚠️
Known behaviour: Same double-send issue as getAll — the 404 branch lacks a return, so the 200 send fires right after. Clients receive the 404 body in practice.
PUT /users/:email Update a user ▼

Updates one or more user fields. Only fields present in the body are modified. If password is included, it is re-hashed before storage.

Path Parameters

ParameterTypeRequiredDescription
email string required Current email address of the user to update.

Request Body

FieldTypeRequiredDescription
name string optional New display name.
email string optional New email. Changes the URL identifier for all subsequent requests.
password string optional New plain-text password. Will be bcrypt-hashed before saving.
Request example
JSON Body
{
  "name": "Alice Dupont"
}
● 200 OK

Returns id, name, and email of the updated user. Password hash is never returned.

JSON
{
  "id":    "664c3e1a2f7d8b001e0a1b2c",
  "name":  "Alice Dupont",
  "email": "alice@marina.fr"
}
● 404 Not Found
JSON
{ "message": "User not found" }
● 500 Internal Server Error
JSON
{
  "message": "Error updating user",
  "error":   "..."
}
⚠️
Real-world note: Updating email changes the URL key used to look up this user on all subsequent GET, PUT, and DELETE calls. If your frontend stores user URLs or email references, update them immediately after a successful email change.
DELETE /users/:email Delete a user ▼

Permanently removes a user account. Returns the deleted user's basic info for confirmation.

Path Parameters

ParameterTypeRequiredDescription
email string required Email address of the user to delete.
Request example
HTTP
DELETE /users/alice%40marina.fr
● 200 OK
JSON
{
  "issues": "deleted",
  "id":     "664c3e1a2f7d8b001e0a1b2c",
  "name":   "Alice Martin",
  "email":  "alice@marina.fr"
}
● 404 Not Found
JSON
{ "message": "User not found" }
● 500 Internal Server Error

Note: the error message label reads "Error updating user" — this is a copy-paste artefact in the source code; it is still returned on a delete failure.

JSON
{
  "message": "Error updating user",
  "error":   "..."
}
🚨
Real-world note: Deleting a user does not revoke any active sessions or tokens. If your authentication layer issues JWTs, ensure they are short-lived or maintain a token blocklist so that deleted users cannot continue to authenticate after removal.
